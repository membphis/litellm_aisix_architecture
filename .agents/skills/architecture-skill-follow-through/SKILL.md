# 架构 Skill 变更推进

## 概述

当用户修改了架构类或规范类 skill，并希望将这些修改转化为下游实现规划工作时，使用此 skill。核心原则是先总结并处理 skill 层边界，在代码规划前稳定相关 skill 变更，然后总结下游实现影响，最后才交接给 `writing-plans`。

## 适用场景

适用于：
- 工作区中存在架构类或规范类 skill 的本地修改，或用户明确指向近期的架构/规范 skill 变更
- 用户希望继续推进实现规划或下游项目规划
- 任务需要在 skill 层变更和代码层规划之间保持清晰边界

不适用于：
- 用户仅编辑 skill 文案，没有下游实现意图
- 当前作用域内没有相关的架构/规范 skill 变更
- 任务已处于纯实现执行阶段且已有批准的计划
- 用户希望直接执行而非规划交接

## 触发边界

此 skill 使用混合触发模式：
- 首先检查本地工作区中架构类或规范类 skill 的 diff
- 仅当用户同时希望进行下游实现或规划跟进时，才激活此流程
- 如果任务仅涉及编辑 skill 文本本身，停在 skill 层，不调用 `writing-plans`

活动变更范围必须锚定在用户指向的具体来源上。
- 如果用户指定了文件、提交、范围、粘贴文本或具体摘要，从那里开始
- 仅当与活动变更组存在直接重叠证据（共享实体名称、共享需求、共享文件引用或用户明确指示）时，才扩展范围
- 不要因为工作区中有其他未提交修改就把无关变更纳入活动组

如果用户已提供具体的变更内容，直接使用该内容，不要强制额外的澄清步骤。

如果用户指向近期的架构/规范 skill 变更，但工作区是干净的且没有现成的具体变更内容，先问一个简短的澄清问题，而不是假设此 skill 不适用。相关变更来源可能是当前分支状态、最近一次提交、粘贴文本或用户提供的摘要。

当来源不是未提交的工作区 diff 时，必须显式检查作用域内的具体变更来源，例如相关提交、提交范围、分支增量、粘贴文本或用户提供的摘要。不要把干净的工作区当作"没有相关变更"的证据。

在本仓库中，最常见的触发是 `.agents/skills/arch/**` 下的本地 diff，下游实现目标通常是 `aisix/`。这些是项目内示例，不是通用假设。

## 必须流程

1. 检测作用域内相关的架构/规范 skill 变更。
2. 判断当前任务是否要求下游跟进。
3. 产出 `Skill Change Intent Summary`。
4. 如果下游跟进仍在作用域内，处理第一次确认边界。
5. 先协调所有相关的架构/规范 skill 编辑。
6. 如果 stabilization 改变了归一化目标，重新生成 `Skill Change Intent Summary` 并在下游跟进仍在作用域内时再次处理第一次确认边界。
7. 产出 `Implementation Impact Summary`。
8. 如果下游跟进仍在作用域内，处理第二次确认边界。
9. 以活动变更组的已解决 `Implementation Impact Summary` 为输入，实际调用 `writing-plans`。

不要跳过、合并或重排这些步骤。

在活动变更组的 `writing-plans` 交接之前，不要开始代码编辑、实现执行、任务排序或详细计划草拟。

最小分组规则：
- 仅当变更贡献于相同的下游规划目标或相同的架构/规范主题时，才归入同一组
- 如果它们会导致不同的下游计划，视为不同组
- 分组不明确时，优先拆分而非合并

如果存在多个不相关的变更组：
- 分割工作流，逐组处理，或
- 问用户先推进哪一组

不要把不相关的架构/规范变更强行合并到一个摘要中。

## 硬性约束

在两份摘要产物都已产出且每道确认边界都已处理之前，不要调用 `writing-plans`。

不要把 skill 层确认和实现影响确认合并成一个问题。

在 `Skill Change Intent Summary` 边界尚未解决之前，不要开始 AISIX 或项目代码规划。

在此 skill 的任何阶段都不要开始实现执行。此 skill 在 `writing-plans` 交接处终止。

使用此 skill 时不要编辑架构/规范 skill 文件，除非用户明确要求你修改它们。此 skill 的默认职责是解释和规划交接，不是修改真相来源的 skill 变更。

默认行为是保留两道显式确认边界。如果用户显式免除某次确认，不要假装确认已经发生。相反，承认该门禁已被用户指令免除，保持产物边界完整，继续下一个必需产物。允许免除确认。不允许跳过产物本身。

在同一活动变更组内，确认边界的显式免除对该边界的后续修订持续有效，除非用户缩小、撤销或替换该免除。

例外：如果 stabilization 实质性地改变了 `Skill Change Intent Summary` 中的归一化目标，则第一次确认边界的任何先前免除无效。变更后的产物与用户免除的产物在实质上不同，因此边界必须重新解决。

仅当用户明确指示不要等待该确认边界时，才视为显式免除。示例：
- `不用确认了，直接继续`
- `不要再问了，直接推进`
- `跳过确认继续走`

仅当用户明确表示当前产物可以接受并继续时，才视为确认通过。示例：
- `确认`
- `这个摘要没问题`
- `是的，按这个继续`

不要把 `继续`、`ok`、`fine`、`sure`、`whatever`、`好的`、`继续`、`可以` 等模糊回复视为免除，除非上下文使免除含义毫无歧义。

不要把上述模糊回复视为确认通过。把任何不属于显式确认或显式免除的回复视为模糊。

如果用户回复模糊，问一个简短的澄清问题，而不是猜测边界是被确认、拒绝还是免除。

如果活动变更组中的 skill 文件在确认边界等待期间被修改，在解决该边界之前重新检查受影响的文件。

当确认边界激活时，停下来等待用户输入后再继续。有效结果：
- 已确认
- 已拒绝
- 被用户显式免除

不要在请求确认的同时继续推进，好像边界已经解决了一样。

## 产物 1：Skill Change Intent Summary

产出一份紧凑摘要，包含：
- 变更的 skill 文件
- 每个文件的变更摘要
- 归一化后的架构/规范意图变化
- `Goals To Drive Forward` 列表

将此产物的第一版视为活动变更组的 pre-stabilization 摘要。它在 stabilization 之后可能变得更精确。

产出此产物前所需的最低证据：
- 检查作用域内所有受影响的 skill 文件
- 比较相关的新旧内容，或以其他方式检查用户所指的具体变更内容
- 不要仅从文件名做摘要

此产物用于回答：
"我们是否正确理解了 skill 层发生了什么变化？"

## 第一次确认边界

当下游跟进仍在作用域内时，默认询问：

`请确认这份 skill 变更摘要是否正确捕捉了你希望推进的架构/规范变更。`

如果用户拒绝，仅修订 `Skill Change Intent Summary`，然后再次请求第一次确认边界。不要进入实现影响阶段。

如果用户显式免除此次确认，记录 skill 层摘要未经确认而是被用户指令免除。不要跳过摘要本身。

如果下游跟进不再在作用域内，必须在呈现 `Skill Change Intent Summary` 供文案或准确性审阅后停止，不强制此确认，不运行 stabilization，不产出 `Implementation Impact Summary`。

请求此确认后，等待用户回复后再继续。

## Skill 层 Stabilization

在第一次确认之后，或对该确认的显式免除之后，确保架构/规范 skill 层在继续之前已内部稳定。

此处的 stabilization 意为对变更 skill 集的分析和协调，不是自动编辑任何 skill 文件。stabilization 期间不要修改 skill 文件，除非用户明确要求你修改。

具体来说：
- 识别受同一变更集影响的相关 skill 文件
- 需要时检查直接相关但未变更的 skill 文件，以理解重叠、边界或冲突
- 将不相关的 skill 变更拆分为不同组，而非强行合并
- 如果分组不明确，先问一个简短的澄清问题
- 检查这些文件之间是否存在重复、冲突或不完整的约束
- 将此视为内部分析检查点，不是额外的用户确认门

不要将下游代码规划视为此 stabilization 步骤的一部分。

如果 stabilization 改变了归一化目标，重新生成 `Skill Change Intent Summary` 并在下游跟进仍在作用域内时再次走第一次确认边界，除非该边界已在活动变更组中被显式免除。

## 产物 2：Implementation Impact Summary

在 skill 层稳定后，将已解决的变更映射到下游工作：
- 受影响的代码区域或项目模块
- 必须跟进的事项
- 可选跟进事项
- 有意暂缓的事项
- 应进入实现计划的事项

产出此产物前所需的最低证据：
- 以活动变更组当前的 `Skill Change Intent Summary` 为基础
- 以该组已稳定的 skill 层解释为基础
- 在声称下游影响之前检查相关代码区域
- 在声称非代码下游影响之前检查相关文档或项目规划区域

不要从粗略检查中过度推断下游范围。如果影响不确定：当事项是可选或低影响时标记为不确定；当事项在实现计划中是必需的时，问一个简短的澄清问题。

不要将此产物变成实现计划。在此阶段你可以识别影响、范围和候选工作项，但不要排序任务、选择执行顺序或草拟计划本身。

此产物用于回答：
"这些是否是接下来应该规划的下游变更？"

Implementation Impact Summary 必须以已解决的 `Skill Change Intent Summary` 为主要来源。原始 diff 仅可用于解决摘要中的歧义。

产物 1 的"已解决"指以下之一：
- 已被用户确认
- 已被用户显式免除

## 第二次确认边界

当下游跟进仍在作用域内时，默认询问：

`请确认这份实现影响摘要是否是正确的下游范围，然后我会调用 writing-plans。`

如果用户拒绝，仅修订 `Implementation Impact Summary` 并再次询问。不要调用 `writing-plans`。

如果用户显式免除此次确认，记录实现影响摘要未经确认而是被用户指令免除。不要跳过摘要本身。

请求此确认后，等待用户回复后再继续。

## 交接

第二次确认边界解决后，实际调用 `writing-plans` 来创建实现计划。

计划必须以活动变更组的已解决 `Implementation Impact Summary` 为基础，不是直接基于原始 skill diff。

"已解决"指以下之一：
- 已被用户确认
- 已被用户显式免除

不要停在口头宣布"即将使用 writing-plans"。此 skill 在 `writing-plans` 实际调用完成之前不算完成，除非用户明确打断或重定向。

## 失败路径

- 作用域内没有相关的架构/规范 skill 变更：退出此 skill。
- 有相关 skill 变更但没有下游规划意图：产出 `Skill Change Intent Summary`，仅作文案或准确性审阅呈现，然后停止。除非用户要求更深入的 skill 层协调，否则不运行 stabilization。
- 第一次确认被拒：仅修订产物 1，然后再次请求第一次确认边界。Stabilization 仅在第一次边界通过确认或显式免除解决后才开始。
- 第二次确认被拒：仅修订产物 2。
- 如果审阅产物 2 时发现产物 1 有误或不完整：回到产物 1，显式修订它，重新运行该活动变更组的 stabilization，然后重新生成产物 2。
- 多个不相关的变更组：逐组处理或问用户先处理哪一组。
- 模糊的确认回复：先问一个简短的澄清问题再继续。
- 部分确认并附带修正要求：视为拒绝并针对当前产物做定向修订。
- 同一产物连续被拒三次：暂停并问用户是继续修订还是停止。
- `writing-plans` 无法调用或调用失败：报告错误并将 `Implementation Impact Summary` 作为独立交付物呈现。

## 项目内适配

保持工作流用词通用：
- 使用 `architecture/specification skill` 而非硬编码某一种命名方案
- 使用 `project codebase` 或 `implementation target` 而非默认写死产品名

允许项目特定示例。在本仓库中，一个常见模式是 `arch-*` skill 变更驱动 `aisix/` 的后续推进。

## 示例

用户：`我刚改了 arch skill，接下来按这些修改继续推进。`

正确行为：
1. 检查作用域内的架构/规范 skill 变更
2. 产出 `Skill Change Intent Summary`
3. 处理第一次确认边界
4. 稳定架构 skill 层
5. 产出 `Implementation Impact Summary`
6. 处理第二次确认边界
7. 调用 `writing-plans`

## 常见合理化借口

| 借口 | 事实 |
|------|------|
| `diff 一目了然，直接开始规划吧。` | 在代码规划前仍然需要 skill 层摘要及其边界处理。 |
| `用户已经要求继续了，我可以跳过一次确认。` | 你可以接受显式免除，但仍然不能跳过摘要产物或其边界。 |
| `只改了一个 skill 文件，架构层已经稳定了。` | 在代码规划前仍需确认归一化目标。 |
| `为了效率我可以把两份摘要合为一份。` | 此 skill 要求分开的架构确认和实现确认边界。 |
| `用户说了不用确认，所以我应该直接进 writing-plans。` | 用户可以免除确认，但你仍必须产出摘要产物并在继续前保持边界显式。 |

## 红旗

如果你即将做以下事情，停下来纠正方向：
- 从 skill diff 直接跳到实现计划
- 只做一次合并确认
- 将 skill 编辑和代码影响映射视为同一产物
- 在实现影响摘要边界尚未解决前调用 `writing-plans`

## 常见错误

- 读完原始 diff 就立即草拟计划
- 把 `.agents/skills/arch/**` 或 `aisix/` 等本地示例当作必须的通用结构
- 在 skill 层边界尚未解决前就开始实现规划
- 把两道确认门折叠成一份摘要

## 快速参考

- 检测相关 skill diff
- 确认下游意图存在
- 产出 `Skill Change Intent Summary`
- 如需要，处理确认边界 1
- 稳定架构/规范 skill 层
- 产出 `Implementation Impact Summary`
- 如需要，处理确认边界 2
- 调用 `writing-plans`
